⚠️
Critical:

All school management tables have RLS that blocks anon. Always use createServiceClient() in server actions/components when reading these. Using the anon browser client silently returns empty arrays.

Authentication

Admin Panel (/admin/*)

Mechanism: SHA-256( ADMIN_PASSWORD + "happyhouse-admin-2025" ) stored in admin_session cookie (httpOnly, 7 days).

Checked in: middleware.ts using Web Crypto API (Edge-compatible).

Login page: /admin/login → server action loginAction() → sets cookie → redirect.

Student Portal (/portal/*)

Mechanism: Supabase Auth session (JWT in cookie, managed by @supabase/ssr).

Login: Student enters phone number (e.g. 0901234567) → converted to 0901234567@hh.local → supabase.auth.signInWithPassword().

Account creation: Admin goes to /admin/school/students/[id] → enters password → server action calls supabase.auth.admin.createUser() → links school_students.auth_user_id.

Checked in: middleware.ts — supabase.auth.getUser() on every /portal/* request.

Public Tests (/test, /t/[id])

No authentication. Student fills name + phone before starting. Results saved to students (upsert by phone) and test_sessions.

Supabase Clients — 3 Variants

// 1. Browser client — client components, portal pages (uses session cookie)
import { createClient } from '@/lib/supabase/client'

// 2. Server client — server components, server actions that read auth session
import { createClient } from '@/lib/supabase/server'

// 3. Service client — bypasses ALL RLS (admin server actions only, NEVER in browser)
import { createServiceClient } from '@/lib/supabase/service'
ClientUse forAuth
createClient() (browser)Portal client components, portal loginUses Supabase session cookie
createClient() (server)Portal server components, getUser()Reads session from Next.js cookies()
createServiceClient()ALL admin server actions, school data fetchingService role — bypasses RLS

Key Architectural Patterns

Server/Client split

// page.tsx (server component)
const data = await createServiceClient().from('...').select()
return <FooClient data={data} />

// FooClient.tsx ('use client')
// handles interactivity, calls server actions on mutation
async function handleSave() { await serverAction(id, formData) }

Test state management

The anonymous test flow (/test, /t/[id]) uses React context + useReducer (lib/test-context.tsx) to share state across landing → test → result pages.

Portal tests (/portal/test/[id]) are self-contained in PortalTestClient — no shared context needed since student identity comes from Supabase Auth.

Question grouping rule

💡
Always call sortQuestionsForDisplay(questions) before rendering a test.

Questions with the same passage_id must be contiguous for split-pane to work (passage left, questions right). The admin may add passage questions out of order; this function re-groups them by first appearance.

Admin layout

All /admin/* pages use h-screen overflow-hidden on the root. Each page is flex flex-col h-full with:

  • Fixed toolbar (shrink-0) — title, filters, search
  • Scrollable body (flex-1 min-h-0 overflow-y-auto) — table or card list
  • Fixed footer (shrink-0) — <Pagination />

Tables use sticky top-0 z-10 on <thead> so headers remain visible while body scrolls.

Optimistic updates

Active/inactive toggles, student add/remove, publish toggles — all use useState to flip the UI before the server responds. On server error the state reverts and toast.error() is shown (sonner).

Feature: Entry Test (public)

Mini: 30 questions (grammar + vocab), 30 min. Full: 95 questions (all sections), 90 min.

Questions are shuffled server-side via loadQuestions() in lib/test-utils.ts: standalone (grammar/vocab) first in random order, then passage groups in random group order but internally stable.

Scoring happens 100% server-side via submit_test() Postgres function — correct answers are never sent to the browser.

Result: total score circle, per-section progress bars (green ≥70%, yellow ≥50%, red below), collapsible "Xem lại câu sai" with each wrong question showing student answer (red) vs correct answer (green).

Feature: Curated Tests

Admin creates a test at /admin/tests/new (deferred save — nothing goes to DB until "Lưu nháp" or "Lưu & Publish" is clicked).

Question picker groups by passage with a "Chọn tất cả" button per group. Admin can reorder questions with ↑↓ arrows.

Publishing creates a shareable link: /t/[testId]. Tests can also be assigned to school classes via class_tests table — enrolled students then see the test in their portal.

Feature: Question Bank

Sections: grammar, vocabulary, reading, listening.

Types: multiple_choice, true_false, fill_blank. Legacy DB values reading/listening treated as multiple_choice — see MULTIPLE_CHOICE_TYPES constant in types/index.ts.

Passages are separate entities. Reading passage = text content. Listening passage = audio URL from Supabase Storage bucket audio (public, max 50 MB, allowed: mp3/m4a/ogg/wav).

Audio upload validates file type by extension (browser file.type can be empty), sanitises filename, and returns the public URL.

Feature: School Management

Terminology distinction: students = entry-test prospects. school_students = enrolled học viên. Completely separate tables.

Many-to-many classes: A student can be enrolled in multiple classes simultaneously via student_classes junction table. Admin adds from existing list (picker with search) or creates new on the class detail page.

Attendance: Marked per class per date. Server actions getStudentsForClass() and getAttendanceForDate() use service client (RLS would block anon).

Portal accounts: Admin sets phone + password → supabase.auth.admin.createUser({ email: phone@hh.local, password }) → links auth_user_id. Student logs in with phone number.

Feature: Student Portal

After login, portal page:

  1. Gets all student_classes enrollments for this student
  2. Gets class_tests for all enrolled class IDs
  3. Deduplicates (same test assigned to multiple classes)
  4. Shows pending (not yet submitted) vs completed tests

Test submission calls submit_test() with p_school_student_id so results are trackable per enrolled student (not just by phone number).

Common Gotchas

🚫
1. RLS silently blocks school tables for anon client

Using createClient() (browser/anon) to read school_students, classes, student_classes, attendance returns empty arrays silently — no error. Always use createServiceClient() in server actions/components for these tables.

⚠️
2. useActionState signature

Server actions used with useActionState need (_: unknown, formData: FormData) — the state param must be first even if unused. Missing this causes a TypeScript overload error.

⚠️
3. Passage questions must be sorted before rendering

Call sortQuestionsForDisplay(questions) from lib/test-utils.ts before passing questions to the test engine. Without it, passage questions can be interleaved with standalone questions and the split-pane UI breaks.

ℹ️
4. Phone → email for portal auth

Student portal login internally maps phone 0901234567 → email 0901234567@hh.local for Supabase Auth. This is invisible to users but must be consistent between account creation and login. See phoneToEmail() in app/admin/school/actions.ts.

✅
5. Optimistic toggles — don't add disabled

Active/inactive toggles flip visually before the server responds. The pattern is useState(initial) → flip immediately → revert on error. Adding disabled during pending breaks the instant visual feedback that makes the UI feel fast.

ℹ️
6. Student picker exclusion is client-side

The class detail student picker fetches ALL students from server, then filters out already-enrolled ones client-side using students.some(e => e.id === s.id). Server-side PostgREST .not('id', 'in', ...) with UUID arrays was unreliable.

Deployment

  1. Connect github.com/ptnam0700/happyhouse to Vercel
  2. Set Root Directory = web in Vercel project settings → General
  3. Add all 5 environment variables in Vercel → Settings → Environment Variables (Production + Preview + Development)
  4. Deploy → get URL (e.g. happyhouse-xyz.vercel.app)
  5. Update NEXT_PUBLIC_SITE_URL to the actual domain → Redeploy (fixes sitemap, OG tags, test share links)

Static pre-rendering: public pages and admin list pages are statically generated where possible. Detail/edit pages are dynamic (server-rendered per request).
